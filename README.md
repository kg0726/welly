## 추천 응답 620ms — N+1과 매 요청마다 전수 조사

사용자별 맞춤 정책 추천 기능을 구현하면서 두 개의 독립적인 성능 문제가 겹쳤습니다.

<img width="3113" height="2112" alt="image" src="https://github.com/user-attachments/assets/f5c9b169-d84f-4ee0-bb74-a017fc9804e1" />

**문제 1. N+1 — 추천 후보마다 연관 데이터를 개별 쿼리로 조회**

추천 로직이 후보 정책 목록을 가져온 뒤 루프를 돌며 지역·혜택 등 연관 테이블을 매번 개별 쿼리로 조회했습니다. 정책 수가 늘수록 쿼리 횟수가 **선형으로 증가**하는 구조였습니다.

**문제 2. 매 요청마다 전체 정책 전수 조사**

추천 점수 산정 로직이 요청마다 전체 정책 데이터를 다시 불러와 스캔했습니다. 동일 조건의 사용자가 반복 요청해도 매번 같은 DB 조회와 연산을 반복했습니다.

Redis 같은 별도 인프라를 구축할 시간과 인적 리소스가 부족했고, 개발 서버 메모리가 제한적이어서 무분별한 캐싱 시 OOM 위험도 있었습니다. 두 문제를 순서대로 해결했습니다.

### N+1 해결 — Fetch Join

Spring Data JPA의 Fetch Join을 적용해 N+1개의 쿼리를 **단 1회로 통합**했습니다. 연관 테이블(지역·혜택)을 한 번의 JOIN 쿼리로 함께 가져오도록 JPQL을 수정했습니다.

### 반복 전수 조사 해결 — Caffeine Cache

인프라 비용이 없는 Spring **로컬 캐시**(Caffeine)를 도입했습니다. 캐시 키를 `지역 + 나이`로 설계했는데, 사용자 ID 단위로 캐싱하면 적중률이 너무 낮기 때문입니다. 복지 정책의 자격 요건은 지역과 나이로 1차 필터링되므로 이 두 값을 키로 쓰면 많은 사용자가 캐시를 공유할 수 있어 **메모리 효율과 적중률**이 모두 높습니다.

<img width="900" height="1800" alt="image" src="https://github.com/user-attachments/assets/492785d0-afbd-4527-aeaa-f42f9db7af28" />


메모리 누수를 막기 위해 `maximumSize`(개수 상한)와 `expireAfterWrite`(시간 만료) 정책을 함께 설정해 힙 사용량을 안전 범위 내로 통제했습니다. 사용자별로 달라지는 관심사 점수 계산은 캐싱 대상에서 제외하고 실시간으로 처리했습니다. 공통 요소인 자격 요건 후보군만 캐싱해 데이터 정확성과 성능을 모두 확보했습니다.

**결과: 수백 회의 쿼리를 1회로 감소, 동일 조건(지역+나이) 재요청 시 DB 접근 없이 메모리에서 즉시 반환 — 평균 응답 속도 620ms → 20ms(배포 서버 기준)**

## AI 추천 사유 60초 — 순차 API 호출의 한계

추천된 정책 목록에 각 정책의 “AI 추천 사유”를 붙여 응답해야 했습니다. 기존 로직은 `for` 문으로 OpenAI API를 순차 호출했습니다. API 1회 호출 평균 1.5초, 40개 정책 처리 시 약 60초 대기 — 사실상 서비스 불가능한 수준이었습니다.

비동기 처리 방식을 고를 때 세 가지 대안을 검토했습니다.

| 방식 | 배제 이유 |
| --- | --- |
| `@Async` | Fire-and-Forget에 최적화. 40개 결과를 모아 한 번에 반환해야 하는 상황에서 동기화 처리가 까다롭고 코드가 복잡해짐 |
| `Future` | `get()` 호출 시 스레드 블로킹 발생, 결과 조합·예외 처리 콜백 기능 부족 |
| `WebFlux` | 전체 스택을 리액티브 패러다임으로 전환해야 함. Spring MVC + JPA 기반 프로젝트에서 단일 기능을 위해 도입하는 것은 오버 엔지니어링 이라고 판단 |

`CompletableFuture`를 선택했습니다. `allOf()`로 **40개 비동기 결과를 한 번에 집계**할 수 있고, Stream API와의 조합으로 가독성을 확보할 수 있으며, 커스텀 스레드 풀을 명시적으로 주입할 수 있습니다.

<img width="2349" height="2016" alt="image" src="https://github.com/user-attachments/assets/ad107443-6059-4fd4-82e5-9a5cff083058" />

`CompletableFuture`의 기본 풀(`ForkJoinPool.commonPool`)을 그대로 쓰면 다른 병렬 스트림 작업과 스레드를 공유해 사이드 이펙트가 발생합니다. AI 전용 `FixedThreadPool(10)`을 별도 빈으로 등록해 리소스를 격리했습니다. 스레드 수 10은 I/O Bound 작업 특성상 코어 수보다 넉넉하게 잡되, 외부 API Rate Limit과 서버 메모리를 고려한 수치입니다.

**결과: 전체 처리 시간이 “모든 요청의 합(Sum)”에서 “가장 느린 요청의 시간(Max)”에 가깝게 단축 — 평균 응답 시간 60s → 6s(배포 서버 기준)**

## 서류 양식 DB 없이 서류 초안 생성 기능 구현

추천된 정책에 필요한 서류 초안을 AI로 자동 생성하는 기능을 구현해야 했습니다. 공공기관별·정책별 요구 서류 양식이 **비표준화**되어 있고 포맷(HWP, PDF, 웹 입력)이 제각각이었습니다. DB 구축에 사용한 외부 API에 서류 양식 파일이나 필드 정보가 없어 템플릿 매칭 방식으로는 **서비스 구축이 불가능한 상황**이었습니다.

모든 양식을 DB화하려는 시도 대신, **LLM의 일반화 능력을 활용**하는 역발상을 했습니다. 사용자가 서류 작성을 힘들어하는 부분은 이름·주소 같은 단순 정보가 아니라 지원동기·계획서 같은 **서사 작성**이라고 분석했습니다.

<img width="850" height="1250" alt="image" src="https://github.com/user-attachments/assets/84d0b8c1-b1f9-4ad0-92c3-f1fd535c1e0c" />

물리적 서식 파일 대신 `정책 정보(Context) + 문서명(Title)`을 조합해 AI가 문서의 성격을 스스로 추론하도록 설계했습니다. 단순 정보는 공란 처리로 정확도를 확보하고, **고관여 정보만 AI가 집중 생성**합니다. 접수처·문의처 등 불필요한 행정 안내는 프롬프트 수준에서 필터링(Negative Prompting)해 실질적 제출용 본문만 추출했습니다.

**결과: 서식 DB가 없는 Zero-Base 상태에서도 모든 정책에 대응 가능한 범용 서류 생성 파이프라인 구축**

## Entity 충돌로 매일 빌드 실패 — Schema-First 전략

N:M 연관관계가 얽힌 복잡한 DB 구조에서 팀원들이 병렬 개발을 진행하자 Entity 코드 충돌과 빌드 실패가 반복됐습니다. 각자 로컬 환경의 `ddl-auto: update` 설정으로 인해 DB 스키마가 제각각으로 diverge하면서 통합 테스트 시 데이터 정합성 오류도 이어졌습니다.

`update` 모드의 근본 문제는 각자의 Entity가 정답이 되어 스키마를 제멋대로 바꾼다는 점이었습니다. 방향을 반대로 바꿨습니다 — DB 스키마를 먼저 정답으로 확정하고 코드가 거기에 맞추도록 강제하는 Schema-First 전략입니다.

팀원들을 설득해 ERD 기반 SQL을 먼저 작성하고 이를 유일한 정답 스키마로 합의했습니다. 이후 `jpa.hibernate.ddl-auto`를 `update` → `validate`로 변경했습니다. `validate` 모드는 애플리케이션 실행 시점에 Entity와 스키마 불일치를 즉시 감지해 배포 전에 차단합니다. 이미 적재된 데이터와 구현된 로직을 유지하면서 컬럼 타입·제약조건을 전수 조사해 Entity Mapping을 재설계하는 작업이 뒤따랐습니다.

**결과: 스키마 불일치로 인한 빌드 실패 0%, 팀원 전원이 동일한 DB 환경에서 개발**

## 공공 API 응답의 비정형성 — AI 데이터 정제 파이프라인

공공 데이터 API 응답 품질이 낮아 그대로 적재하면 정렬·필터링 기능 구현이 불가능했습니다.

**문제 1. 추상적 금액 표기**

‘지원 금액’ 필드가 “최저시급”, “생활임금” 등 텍스트로 제공되어 금액순 정렬이 불가능했습니다. 동일한 의미의 표현도 띄어쓰기·동의어 차이로 수십 가지가 섞여 있었습니다.

**문제 2. 신청일 누락 및 비정형 포맷**

‘신청 기간’ 데이터가 Null이거나 `yyyymmdd ~ yyyymmdd` 포맷을 따르지 않아 마감 정보를 정확히 제공할 수 없었습니다.

**문제 3. 복수 지역 코드 — 1:N 구조의 한계**

초기에 `policies` 테이블에 `region_code` 컬럼 하나만 두었으나, API 응답이 `"30110,30140"` 처럼 CSV 형태의 복수 지역 코드를 반환해 데이터 원자성이 위배됐습니다.

<img width="3336" height="3162" alt="image" src="https://github.com/user-attachments/assets/cb54c268-8655-411a-ad8f-45efc2c45e62" />

추상적 금액은 AI를 두 번 호출해 **외부 변수 추출 → 동의어 제거 → Dictionary Mapping으로 구체적 금액으로 환산** 후 적재했습니다. 신청일은 온통청년 API를 2차 호출해 상세 요강을 AI로 분석해 유효한 날짜를 추출했습니다. 복수 지역 코드는 `policy_regions` 매핑 테이블을 신설해 1:N → N:M으로 구조를 확장하고 파이썬 파싱 로직(`parse_region_codes`)을 구현했습니다. 별도로 약 1,500건 표본을 AI로 분석해 공통 서류 상위 30개(등본, 초본 등)를 마스터 테이블로 분리하고 N:M 중계 테이블로 연결했습니다.

**결과: 텍스트로만 존재하던 금액 데이터를 수치화해 금액순·마감 임박순 정렬 기능 구현, 복수 지역 정책 누락 없이 노출**
